## Optimistic Concurrency Control

no lock, overcome deadlock

- Phase 1: buffer write set and read set
- Create a critical section: validate serializability in read set when commiting
- ok ? commit : abort

Validate with TX **version** rather than value

```python
# read
val_a = read(A)
read_set.add(val_a)

# validate and commit
for d in read_set:
   if d has changed:
      abort()
for d in write_set:
   write(d)
```

**Problem**

atomic in buffering

before or after in committing

**Solution**

Still use 2PL in critical stage, and use sort to pre-order

```python
    for d in sorted(read-set + write-set):
        d.lock()
    for d in read-set:
        if d has changed:
           abort()
    for d in write-set:
        write(d)
    // release the locks
```

Optimize: read-set is implicitly locked, so read-set lock can be saved

ensure read-set cannot be locked

```python
    for d in sorted(write-set):
        d.lock()
    for d in read-set:
        if d has changed or d has been locked:
           abort()
    for d in write-set:
        write(d)
    // release the locks
```

**Advantages**

- Faster: OCC more reads and less writes
- Faster: OCC less lock

### Paxos

make sure only one writer

Machines keep record of:

- Mn: my proposal number
- Va: accepted value of Na (Value agreed on)
- Na: highest proposal number accepted
- Nh: highest proposal number seen

**Phase 1**

- requested proposer becomes leader, leader create **proposal N** and send to Quorum
- Accepter judges:
  - reject if ID <= previous proposal
  - if ID > previous ones, reply <promise, N, V=null>
  - Promise to abandon ID < N

**Phase 2**

- Leader check received promise, if enough(half+1)
  - choose a V, from Va or random is all Va == null
  - send **<accept, N, V>**
- Accepter judges again:
  - if the promise is gone, ignore
  - register V and send accepted message
